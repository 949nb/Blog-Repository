理解 Session 和 Cookie 、token

在我第一次面试的时候，面试官就问到了我token的事情。但是只是知道Cookie会随着浏览器的带回给服务器，这个算是http管理状态的一种方法。也仅仅只是知道这个概念而已，所以我只是回答了Cookie，看着面试官疑惑的眼神，我就知道我需要好好了解一下token，也就是前端鉴权的方式。

发展史

  • 在很久以前，网页的作用仅仅只是浏览文档，也不需要记录谁浏览过这个文档。所以HTTP的作用仅仅是一个请求加响应就可以了。每一次请求都是一次新的HTTP请求，也没有什么用户登陆注册，每一个请求都是全新的。

  • 渐渐的、在线购物网站的兴起、需要登录的网站不断的增加，HTTP协议就面临一个问题，就是需要记录状态。他需要知道这个人是否登陆了，是否是特定用户登陆了网站，或者请求了服务器。也就是说需要把每一个人都区分开。这就是一个难题，因为HTTP是无状态的协议。所以最开始的办法就是给大家发一个Session ID，每个人都不一样，每次对服务器发起请求的时候，都捎带把这个Session ID发过来。这样服务器就能区分开谁是谁了～

    如果没有这个Session ID，想一下

  • 客户端只保存自己的Session ID就好了。问题在于，服务器需要保存每个人的Session ID，一旦访问量变大，成千上万，甚至几十万个人访问的话，对于服务器就是一个巨大的开销，严重的限制了服务器的扩展能力。比如说两个机器组成了一个集群，小S通过机器A登陆了系统，那Session ID 就会被保存在机器A上，假设小S下一次的请求被转发到了机器B上怎么办？机器B就没有这个用户的Session ID。

    这个时候就需要将小S的请求一直在机器A上，但是这样也会出现问题，万一机器A挂了，那小S的请求也会被转到机器B上。

    还有一个方法就是两台机器都复制Session ID，

    还有几种解决办法，但是这个小小的Session ID也成为了服务器的负担了。

  • 接下来就有人想要解决这个烦人的Session ID了，这些成千上万的Session ID只让客户端去存储就好了。如果不储存这些Session ID就要想办法去验证这些客户端发来的ID的有效性，如果想减轻服务器的负担，就要想出有效的办法来验证Session ID的有效性。

token

  • 如果 小S 已经登陆了系统，我给他发一个令牌(token)，带上小S的 User ID，下一次小S再次通过HTTP请求访问我的时候，这个token通过HTTP Hearder带过来，这个方案是不行的，因为这样的验证方法任何人知道了用户ID之后都是可以伪造的，起不到验证的作用。

  • 因为上面的把user id送了过来保证了鉴权信息，所以用一个只有服务器端知道的密钥加密这个数据，发给客户端，然后客户端每次请求的时候都将数据反回来，服务器利用唯一的密钥解码，就可以保证这个验证数据不被轻易伪造。

    • 在原文中写的是给数据做签名，阮一峰-什么是数字签名看了这篇文章明白了什么是数字签名，和我想的是一样的 只不过对信息加密的不是服务器的私钥，而是证书中心给的数字证书，而浏览器有自带的权威认证中心的公钥。这就是HTTPS网页加密的一部分，这篇文章说的token用签名应该不分公钥私钥，只是服务器有这样一个可以解锁的密钥，对数据进行加密。
    • 这样相应给客户端的token服务器就可以不保存，直接对发送过来的token 进行特定算法和自己密钥的解密。这样和token中的签名做个比较，如果一样，就说明是这个用户进行登陆，如果不一样数据一定被篡改过，返回 对不起，没有认证。

什么是HTTP？

HTTP协议用于客户端和服务器端之间的通信

  • HTTP协议和TCP/IP协议中的大多数协议一样，用于客户端和服务器之间的通信。用HTTP协议能够分辨出哪边是客户端，哪边是服务器端。（根据报文的信息）
  • HTTP协议规定，请求从客户端发出，最后服务器端响应该请求并返回。换句话说，肯定是先从客户端这边开始建立通信的，服务器端在没有接收到请求之前不会发送响应。
  • 下面是一个具体的范例：

  • 请求报文 是由请求方法、请求URL、协议版本、可选的请求首部字段和内容实体构成的。在我理解就是四部分：
      1. 方法、URL、协议版本号
      2. 首部字段
      3. 空格
      4. 参数实体
  • 响应报文 和请求报文是一样的，都是四部分组成 唯一不同的就是第一部分的版本号，改成了响应回来的状态码。

HTTP是不保存状态的协议

HTTP本身是无状态协议，也就是说在HTTP这个级别，协议对于发送过的请求或响应都不会做持久化处理，协议本身不具备保存之前发送过的请求和响应的功能。

HTTP虽然是无状态协议，但是为了实现期望的保持状态的功能（例如：鉴权），于是引入了Cookie技术。有了Cookie在用HTTP协议通信，就可以管理状态了。

HTTP 请求方法

  • GET：获取资源（给我那个资源！）

  • POST：传输实体主体

  • PUT：传输文件(存在安全问题，任何人都可以向服务器传输文件)

  • HEAD：获得报文首部（告诉我通讯情况！）

  • DELETE：删除文件(和PUT一样 存在安全问题)

  • OPTION：询问支持的方法？

  • TRACE：追踪路径

    这些请求方法 除了POST和GET，其他的都不常用。以后自己会写服务器了，写一个服务器玩一下这些方法再。还有就是GET和POST其实都能传输实体的主体，区别就是GET的默认参数是在URL的后面的，而POST是在请求报文的第四部分。其中还有很多的CENTENT-TYPE可以选择。这个后面再写。

首先说一下印象比较深的TCP/IP协议族

计算机和网络需要互相通信，就需要有一个规则，那么这个规则就叫做协议。协议中有很多内容，其中就包括我建站想要了解的客户端和服务器建立通信的顺序，以及web网页显示需要处理的步骤，等等。。

TCP/IP有一个分层管理，很重要。他把层次分为：应用层、传输层、网络层和数据链路层

  • 应用层
    • 应用层给用户提供了通讯时需要的应用，比如说DNS服务(域名系统)，FTP服务(文件传输协议)，HTTP协议等。
  • 传输层
    • 传输层对上层应用层提供两台计算机之间的数据传输，比如TCP协议、UDP协议（这个没听说过 0-0）等。
  • 网络层
    • 网络层又叫网络互联层，它用来处理网络上流动的数据包。数据包是网络传输时的最小数据单位，在一台服务器或者计算机，与多台计算机或者网络设备进行数据传输的时候，它起的作用就是选择一条传输线路，IP协议在这一层。
  • 链路层
    • 用来处理连接网络的硬件部分，包括网络适配器，网卡，驱动，光纤等物理可见的部分。

客户端到服务端 是这样的顺序 应用层 ==> 传输层 ==> 网络层 ==> 链路层。发送端从应用层往下走，接收端从链路层往上走。

在层与层之间传输的时候，每经过一层必然会被打上该层的首部信息。反之，则会一层一层的把对应的首部去掉。

与HTTP关系密切的协议：IP、TCP、DNS

  • IP协议
    • IP协议位于网络层，几乎所有的使用网络的系统都会用到IP协议。IP协议包含了两个比较重要的概念：IP地址、MAC地址。IP地址指名了节点被分配到的地址，MAC地址是指网卡所属的固件地址。IP地址可以和MAC地址进行配对，IP地址可更换，但MAC地址基本不会更换。利用一种叫做ARP的协议，通过IP地址查找MAC地址。
  • TCP协议
    • TCP协议位于传输层，为了准确的讲数据送达目标处，TCP协议采用了三次握手的策略。
  • DNS域名解析服务
    • DNS用于应用层，计算机可以被赋予IP地址，也可以被赋予主机名字和域名。用户通常使用域名来访问对方的计算机或者服务器，而不是直接通过IP地址来访问。所以，通过DNS域名解析服务，就可以通过域名来查找IP地址。

这些协议与HTTP之间的关系是怎么样的呢？

如上图所示，客户端输入域名，先是应用层DNS开始解析域名给出IP地址，通过HTTP报文来表达这次请求具体想要干什么？然后传输层TCP协议将请求报文分割成段，运送到网络层IP协议寻找对方的MAC地址，最后通过链路层 一遍中转一边传输。。到达目标之后TCP协议首先重组被分割的报文，然后服务器响应的结果也是一样通过这种TCP/IP的协议传输。

URL/URI

这两个比较好理解，URL就代表统一资源定位符，也就是浏览网页所需要输入的网页地址。URI就代表统一资源标识符，说的意思是某个协议方案表示的资源的定位标识符。
